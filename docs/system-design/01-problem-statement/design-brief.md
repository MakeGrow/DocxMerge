# Design Brief: DocxMerge

## 1. Project Overview

**Package**: `mkgrow/docx-merge`
**Type**: Framework-agnostic PHP 8.1+ Composer library
**Purpose**: Merge DOCX documents by substituting `${MARKER}` placeholders in a template with content extracted from other `.docx` source files.

## 2. Scope

DocxMerge replaces a monolithic 7,357-line Laravel-coupled class (`DocxMergeByMarkers`) with a clean, testable, framework-agnostic library. The library handles the complete OOXML merge pipeline: marker detection, content extraction, style/numbering/relationship remapping, media and header/footer copying, section properties application, and output validation.

### In Scope

- Substitution of `${MARKER}` placeholders in a DOCX template with content from source DOCX files.
- Merging of styles, numbering definitions, relationships, media files, headers, and footers.
- Section properties preservation (page layout, margins, orientation) including intermediate sections.
- Explicit marker-to-section association for multi-section source documents.
- Detection of markers fragmented across multiple `w:r`/`w:t` elements by Word.
- ID remapping (relationship IDs, style IDs, numbering IDs, drawing object IDs, bookmark IDs) to prevent collisions.
- Post-merge validation of document integrity.
- Source document caching for performance when multiple markers reference the same file.

### Out of Scope

- Editing textual content within paragraphs (only block-level substitution).
- Format conversion (PDF, HTML, ODT).
- VBA macro manipulation.
- Password-protected documents.
- Comments, footnotes, and endnotes merging from source documents.
- Charts, diagrams, and OLE objects (deferred to v2).
- Streaming/chunked processing for very large documents.

## 5. Initial Assumptions

1. Templates are valid DOCX files generated by Microsoft Word or LibreOffice. Corrupted templates are rejected during validation.
2. Markers follow the pattern `${NAME}` where NAME matches `[A-Z_][A-Z0-9_]*`. The pattern is configurable.
3. Source documents are valid DOCX files. Invalid sources are rejected with typed exceptions.
4. The library runs in a single-threaded PHP process with sufficient memory for the documents being processed.
5. The caller has write access to the output directory.
6. The reference implementation's core merge algorithm is functionally correct for the majority of OOXML scenarios, with known bugs documented and addressed by design.

## 6. Key Design Decisions

1. **Decomposition**: ~20 single-responsibility classes organized in 4 layers (Public API, Orchestration, Domain Services, Infrastructure).
2. **Stateless services**: All mutable state encapsulated in `MergeContext`, created fresh per operation.
3. **Explicit marker-section association**: `MergeDefinition` DTO with optional `sectionIndex` replaces fragile heuristics.
4. **Content-hash style comparison**: SHA-256 hash of normalized XML for O(1) lookups instead of O(N*M) pairwise comparison.
5. **Defense-in-depth numbering**: Both ordered insertion and post-merge resequencing guarantee OOXML-compliant `numbering.xml`.
6. **DOM node list over XML strings**: `ExtractedContent` carries `list<DOMNode>` for direct `importNode()` without re-parsing.

## 7. Constraints

- No classes from Laravel, Symfony, or any framework in `src/`.
- PSR-3 `LoggerInterface` injected via constructor with `NullLogger` default.
- All exceptions extend `DocxMerge\Exception\DocxMergeException`.
- Every class is `final`. Every property and parameter is explicitly typed.
- PHPDoc on every class, interface, and public method.
- All documentation and code comments in English.
